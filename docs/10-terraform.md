# Terraform (via terranix)

Network/DNS infrastructure that lives outside NixOS (the UniFi controller,
Cloudflare DNS) is described declaratively in Nix using **terranix**, which
renders to standard Terraform JSON (`config.tf.json`) and is applied with
plain `terraform`. This keeps the same "everything is Nix" discipline as the
rest of the repo while still using Terraform's state/plan/apply model for
resources that aren't NixOS hosts.

## 1. Why terranix, not raw `.tf`/HCL

- `terraform/<stack>/default.nix` is authored in Nix and exposed as a flake
  package (`packages.terranix-unifi`, `packages.terranix-cloudflare` in
  `flake.nix`), so it's built/evaluated the same way as everything else in
  the repo, and gets `nix flake check` coverage.
- `terraform/<stack>/config.tf.json` (the rendered output) is gitignored —
  it's a build artifact, regenerated by `make tf-*` before every
  plan/apply.
- State lives in `terraform/<stack>/` per stack (also gitignored —
  `.terraform/`, `*.tfstate*`).

## 2. Stacks

### `unifi` (`terraform/unifi/default.nix`)

Provider: `paultyng/unifi ~> 0.41.0`, talking to the UDM Pro at
`https://10.1.1.1` (`allow_insecure = true` — self-signed cert).

| Resource | Purpose |
|---|---|
| `unifi_network.default` | Default LAN, `10.1.1.0/24`, DHCP `10.1.1.6`–`.254`, DNS `10.1.1.20` |
| `unifi_network.cameras` | VLAN 2, `192.168.3.0/24` | DHCP `192.168.3.6`–`.254` |
| `unifi_network.iot` | VLAN 3, `192.168.2.0/24` | DHCP `192.168.2.6`–`.254` |
| `unifi_firewall_group.dns_ports` | Port group: 53, 853 |
| `unifi_firewall_group.dns_resolvers` | Address group: `10.1.1.20` |
| `unifi_firewall_rule.allow_resolver_dns` | Allow the resolver host out on 53/853 |
| `unifi_firewall_rule.block_all_other_dns` | Drop every other LAN→WAN DNS attempt |

Variable: `unifi_password` (sensitive). Provider auth uses username `admin` +
this variable.

These network/VLAN definitions are the source of truth that
`jupiter.dns.allowedNetworks` (on `lenovo`) must be kept in sync with — see
[05-networking.md §1](05-networking.md#1-subnets--vlans) and the comments in
both files / `CLAUDE.md`.

### `cloudflare` (`terraform/cloudflare/default.nix`)

Provider: `cloudflare/cloudflare ~> 4.0`.

| Resource/data | Purpose |
|---|---|
| `data.cloudflare_zone.jupiter_au` | Looks up the `jupiter.au` zone |

Variable: `cloudflare_api_token` (sensitive). This stack is currently

## 3. Render → plan/apply flow (`Makefile`'s `tf-run` macro)

```make
define tf-run
  nix build .#terranix-$(1) --no-link --print-out-paths | xargs -I{} install -m600 {} terraform/$(1)/config.tf.json
  cd terraform/$(1) && terraform init -input=false
  sops exec-env secrets/secrets.yaml 'cd terraform/$(1) && \
    TF_VAR_unifi_password="$$unifi_password" \
    TF_VAR_cloudflare_api_token="$$cloudflare_api_token" \
    terraform $(2)'
endef
```

1. **Render** — build the terranix package, copy the resulting JSON into `terraform/<stack>/config.tf.json` (mode 600 — it may contain structure that shouldn't be world-readable).
2. **Init** — `terraform init -input=false`.
3. **Plan/apply** — run the requested Terraform subcommand with both provider credentials exported as `TF_VAR_*`, sourced from `secrets/secrets.yaml` via `sops exec-env` (never written to disk).

Invoked via:

```bash
make tf-plan-unifi
make tf-apply-unifi
make tf-plan-cloudflare
make tf-apply-cloudflare
```

`tf-apply-cloudflare` will fail today because `cloudflare_api_token` is not
yet present in `secrets/secrets.yaml` — add it first with
`sops secrets/secrets.yaml` (see
[07-secrets-management.md](07-secrets-management.md)).

## 4. Required dev tooling

`terraform`, `sops`, and `age` are provided by the repo's dev shell
(`shell.nix`, `nix develop`) — no need to install them separately.
